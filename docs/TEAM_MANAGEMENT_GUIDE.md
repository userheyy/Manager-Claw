# OpenClaw 小队团队管理指南（v1）

## 1. 目标

用一个主控 Agent（主虾）+ 多个执行 Agent（小虾），实现任务自动拆解、认领、执行、验收与回收，减少人类参与成本。

---

## 2. 角色设计

### 2.1 主虾（Orchestrator）

职责：
- 接收人类需求（文字/截图）
- 拆解为可执行 Issue / 子任务
- 写入任务池（`TASK.md` 或任务中心）
- 审核 PR、合并、更新积分和状态

约束：
- 只做调度和评审，不直接承担全部开发实现
- 每次仅推进一轮决策，不并发多轮调度

### 2.2 小虾（Worker）

职责：
- 从任务池认领一项任务
- 读取对应 Issue 详情并执行
- 按规范提交产物（代码/PR/报告）
- 回写结构化结果

约束：
- 单轮只做一件事
- 不问“可以开始了吗”，默认拿到任务就执行
- 必须按结构化格式回报

---

## 3. 任务生命周期

状态流转：
`todo -> claimed -> in_progress -> review -> merged/done`  
异常流转：
`in_progress -> blocked`（说明阻塞原因与所需输入）

最小字段：
- `task_id`
- `title`
- `owner`
- `status`
- `issue_url`
- `priority`
- `score`
- `updated_at`

---

## 4. 调度协议（强约束）

### 4.1 单轮指令

每次只下发一步，禁止一条消息塞多个目标。

### 4.2 结构化回包

统一要求只回 JSON：

```json
{
  "task_id": "T20260304-001",
  "status": "ok|blocked|failed",
  "summary": "一句话结论",
  "artifact": "产物路径或空",
  "next_input": "下一步需要的输入"
}
```

### 4.3 串行优先

默认 A -> B -> C 串行。只有明确可并行且无依赖时才并发。

---

## 5. 记忆策略（重点）

结论：牛马型小队不做“厚记忆”，做“薄记忆 + 强任务池”。

规则：
- `MEMORY.md` 只保留长期、稳定、跨项目有效信息
- 项目态信息全部放任务池和 issue，不放长期记忆
- 定期清理过时记忆，避免旧项目污染新项目

---

## 6. 容器隔离策略

为什么用容器 Worker：
- 文件系统隔离，避免互相污染
- 端口隔离，避免本地测试冲突
- 失败可快速销毁重建，恢复成本低

例外：
- 需要宿主机高权限（如 CI/CD、Docker 管理）的 Agent 可以在宿主机运行，但必须单独权限管控。

---

## 7. 激励机制（积分）

建议：
- 完成任务 +X 分
- 高质量通过评审 +Y 分
- 无效提交 / 回滚扣分

用途：
- 作为行为引导，不作为唯一目标
- 与任务难度、返工率联动

---

## 8. 节奏设计（Ralph Loop）

建议循环周期：
- 每 5 分钟唤醒一次 Worker
- 每轮最多处理 1 个任务
- 轮末必须写结构化回报

目标：
- 通过高频小步收敛复杂需求
- 避免一次性大任务失控

---

## 9. 落地清单（你当前项目）

- [ ] 主虾拆解器：需求 -> Issue 列表
- [ ] 任务中心：可认领 / 可回收 / 可追踪
- [ ] Worker 执行器：读取任务 -> 执行 -> 回报
- [ ] 评审合并器：PR 审核与积分更新
- [ ] 污染防护：记忆清理 + 项目隔离

---

## 10. 反模式（避免）

- 给 Worker 一个超长 GOAL.md 然后期待它“自主完成全部”
- 把项目细节长期写进 MEMORY.md
- 让多个 Worker 在同一宿主机直接抢端口
- 不定义结构化回包，导致主虾无法自动判定下一步
